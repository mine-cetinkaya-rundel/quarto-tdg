7  Quarto and R

When you include R code cells in your document, Quarto uses the knitr engine to execute them. The basics of code cells, including syntax and common options like echo and include, are covered in Chapter 4. This chapter covers the features specific to R and knitr.

7.1 Output

7.1.1 Inline code

In addition to code cells, you can embed R code directly within your markdown text using inline code, inserting computed values into your narrative without manually copying them.

The syntax uses single backticks with {r} immediately after the opening backtick:

The mean temperature was `{r} mean(airquality$Temp)` degrees.

We recommend that you avoid complex function calls or operations that do significant work in inline code and instead compute values in a code cell and then reference them inline:

avg_temp <- mean(airquality$Temp)
The mean temperature was `{r} avg_temp` degrees.

This renders as: The mean temperature was 77.8823529 degrees.

By default, inline code produces plain text. To generate markdown-formatted output, wrap the result in I():

The result is `{r} I("**significant**")`.

This produces bold text in the rendered output.

NoteInline code in YAML

Inline code cannot be used directly in YAML strings, such as titles or figure captions. For dynamic captions, use the cross-reference div syntax instead (see Section 12.10).

7.1.2 Data frame printing

Quarto provides the df-print option to control how data frames appear in your output, and it can be set at the document-level in the YAML header:

---
title: "My Analysis"
format:
  html:
    df-print: paged
---

The default option uses the standard S3 print method, i.e., tables appear as they would in the R console. The tibble option prints the table as a plain text tibble, which is equivalent to wrapping the resulting table with tibble::tibble(). The kable option applies knitr::kable() and supports various output formats, including HTML, PDF, and Word. In HTML documents, the paged option enables interactive tables with pagination.

7.1.3 Graphics devices

When R generates a plot, knitr uses a graphics device to save it to a file, which is then included in the output. The default device is png, and it works well in most situations. You may want to change it to improve text rendering, achieve scalable output, or ensure PDF compatibility.

You can set the graphics device with the dev option, either at the document level or for individual cells:

---
knitr:
  opts_chunk:
    dev: ragg_png
---

Or for a single cell:

```{r}
#| dev: svglite
ggplot(mtcars, aes(x = wt, y = mpg)) +
  geom_point()
```

Common choices include the following:

  • The png device is the default.
  • The ragg_png device creates higher-quality PNG images for better text rendering and anti-aliasing. Generating images figures with this device requires the ragg package.
  • The svglite device creates scalable HTML output in vector format. Figures created with this device are sharp at any zoom level and they require the svglite package.
  • The pdf device generates figures in standard PDF format. It is the default vector device for LaTeX/PDF output.
  • The cairo_pdf device generates figures in PDF format with Unicode text. It is especially useful for handling non-Latin characters and special symbols.
TipAutomatic device selection for PDF

The R option knitr.graphics.auto_pdf (shown in Section 7.2.1) tells knitr to automatically use a PDF graphics device when rendering to PDF output and a PNG device otherwise. That way, you don’t need to set dev differently for each format.

While not knitr-specific, the fig-format option is also worth knowing about alongside dev. It works across all Quarto engines:

---
format:
  html:
    fig-format: svg
---

Use this when you only need to change the output file format without customizing the specific graphics device. When both dev and fig-format are set, dev takes precedence.

7.1.4 Multiple figures from a code cell

A single R code cell can produce multiple plots. By default, each plot appears as a separate figure in the output. Use fig-cap and fig-subcap to arrange them as labeled subfigures.

You can provide fig-cap as the overall caption and fig-subcap as a list of individual subcaptions:

```{r}
#| label: fig-mtcars-panels
#| fig-cap: "Motor Trend car data"
#| fig-subcap:
#|   - "Weight vs. fuel efficiency"
#|   - "Displacement vs. fuel efficiency"
#| layout-ncol: 2
library(ggplot2)

ggplot(mtcars, aes(x = wt, y = mpg)) +
  geom_point()

ggplot(mtcars, aes(x = disp, y = mpg)) +
  geom_point()
```

This produces two side-by-side subfigures, each with its own caption, under a shared parent caption. The individual panels are cross-referenceable as @fig-mtcars-panels-1 and @fig-mtcars-panels-2, while @fig-mtcars-panels refers to the group.

Additionally, the layout-ncol option controls how many columns the subfigures occupy. For more complex arrangements, you can use the layout option to specify custom grids:

```{r}
#| label: fig-custom-layout
#| fig-cap: A figure with custom layout
#| fig-subcap:
#|   - Plot A
#|   - Plot B
#|   - Plot C
#| layout: [[1,1], [1]]
plot(1:10)
plot(10:1)
plot(1:20)
```

The layout: [[1,1], [1]] specification places the first two plots side by side in the first row, with the third plot spanning the full width of the second row. For a full explanation of the layout syntax, see Section 12.15.

7.1.5 Interactive output

Many R packages produce interactive HTML output—plots, tables, maps—through the htmlwidgets framework. Load the package and call the function as you would in an interactive R session. For example, you can use code cells that produce interactive plots with plotly:

```{r}
#| eval: false
p <- ggplot(mtcars, aes(x = wt, y = mpg, color = factor(cyl))) +
  geom_point()

ggplotly(p)
```

Or interactive tables with DT:

```{r}
#| eval: false
DT::datatable(mtcars, options = list(pageLength = 5))
```

Or interactive maps with leaflet:

```{r}
#| eval: false
leaflet() |>
  addTiles() |>
  addMarkers(lng = -73.99, lat = 40.73, popup = "New York City")
```

It is important to note that interactive widgets only work in HTML output formats. When rendering to PDF or Word, widgets will either produce a static fallback image or an error, depending on the package. You can use knitr::is_html_output() (see Section 7.2.3) to conditionally include widgets only in HTML output.

NoteWidget dependencies and self-contained documents

Each HTML widget brings its own JavaScript and CSS dependencies. In self-contained HTML documents (embed-resources: true), these dependencies are bundled into the file, which can increase its size. If your document uses many different widget packages, consider using the default non-self-contained output and hosting the result as a directory.

7.2 Configuration

7.2.1 Execution options

Chapter 4 introduced cell-level options like echo and include. Many of these options can also be set at the document level to apply to all code cells.

7.2.1.1 Document-level options

Set execution options for the entire document using the execute key in your YAML header:

---
title: "My Report"
execute:
  echo: false
  warning: false
---

Common document-level execution options include the following and take on values true or false, unless specified otherwise:

  • eval: Whether to execute code cells
  • echo: Whether to display source code
  • output: Whether to include results (can also be set to asis)
  • warning: Whether to include warnings
  • error: Whether to include errors (without halting)
  • include: Whether to include any output at all

Additionally, fig-width and fig-height can be set at the document level to define default figure dimensions in inches.

7.2.1.2 Knitr-specific options

The knitr engine has additional options that you can set using the knitr key. These map directly to knitr’s chunk options:

---
title: "My Document"
knitr:
  opts_chunk:
    collapse: true
    comment: "#>"
    R.options:
      knitr.graphics.auto_pdf: true
---

R.options temporarily sets R options via options() before each code cell executes. The original values are restored afterward.

7.2.1.3 Cell-level options

Individual code cells can override document-level settings. The knitr engine supports a large set of cell-level options.

  • Some options control code appearance:
    • code-fold: Collapse code into an expandable section
    • code-summary: Text shown for collapsed code (e.g., “Show code”)
    • code-line-numbers: Add line numbers to code
  • Some control the output:
    • results: How to display text results (markup, asis, hold, hide)
    • collapse: Merge code and output into a single block
    • comment: Prefix for output lines (default "##")
  • Some are specific to code cells that generate figures:
    • fig-cap: Caption
    • fig-alt: Alternative text
    • fig-align: Alignment: left, center, or right
    • fig-dpi: Resolution in dots per inch
    • fig-width and fig-height: Figure dimensions in inches
    • out-width and out-height: Output dimensions as percentage (80%), pixels (400px), or page metrics (0.5)
  • Finally, some relate to caching:
    • cache: Enable caching for the cell
    • dependson: Specify cells this one depends on

For a complete list of knitr cell options, see the Quarto reference.

7.2.1.4 Raw markdown output

Sometimes you want your R code to produce raw markdown that Quarto processes directly, rather than being wrapped in output containers. Use output: asis for this:

```{r}
#| output: asis
cat("## A Dynamic Heading\n\n")
cat("This paragraph was generated by R.\n")
```

One common use case is programmatically generating sections in a loop.

7.2.2 Parameters

Parameters let you create variations of a document without editing the source—for example, generating the same report for different regions, time periods, or scenarios.

You can define parameters in your YAML header using the params key:

---
title: "Regional Sales Report"
params:
  region: "North"
  year: 2026
  show_details: true
---

Parameters can be accessed in code through the params list:

```{r}
#| echo: false
sales_data |>
  filter(region == params$region, year == params$year)
```
This report covers the `{r} params$region` region for `{r} params$year`.

It is also possible to override parameters from the command line using the -P flag:

quarto render report.qmd -P region:South -P year:2025

For multiple parameter sets, you can create a YAML file with the values:

params-south.yml
region: "South"
year: 2025
show_details: false

Then reference it when rendering:

quarto render report.qmd --execute-params params-south.yml

7.2.3 Conditional execution

Sometimes you need code to behave differently depending on the output format. Quarto provides several ways to access format information from your R code.

The knitr package provides functions to check the output format:

```{r}
if (knitr::is_html_output()) {
  # Code for HTML output only
}

if (knitr::is_latex_output()) {
  # Code for PDF output only
}
```

You can also check the specific Pandoc output format:

```{r}
knitr::pandoc_to()
# Returns e.g., "html", "latex", "docx"
```

You can also use Quarto environment variables for conditional execution. For more detailed format information, Quarto sets the QUARTO_EXECUTE_INFO environment variable during rendering:

```{r}
info <- jsonlite::fromJSON(Sys.getenv("QUARTO_EXECUTE_INFO"))
format_name <- info$format$identifier$`base-format`

if (format_name == "html") {
  # HTML-specific code
}
```

This provides finer-grained format details than knitr::pandoc_to(), such as sub-format identifiers and format options.

7.3 Rendering

7.3.1 Rendering from R

The quarto R package provides functions to render documents directly from R, without using the terminal. This package is optional — you don’t need it to use Quarto with R. It’s a convenience for those who prefer working entirely within the R console.

You can use quarto_render() to render a document:

library(quarto)

quarto_render("report.qmd")
quarto_render("report.qmd", output_format = "pdf")

For iterative work, quarto_preview() starts a live preview that updates as you save changes:

quarto_preview("report.qmd")

# When done:
quarto_preview_stop()

You can also pass parameters directly when rendering with quarto_render() using the execute_params argument:

quarto_render(
  "report.qmd",
  execute_params = list(region = "South", year = 2025)
)

7.3.2 Rendering R scripts

You can render R scripts (.R files) directly with Quarto, without converting them to .qmd files first. This uses syntax based on knitr::spin().

  • Lines starting with #' are treated as markdown
  • Lines starting with #| are code cell options
  • All other lines are R code

Here’s an example script that includes both markdown and code cells:

analysis.R
#' ---
#' title: "Penguin Analysis"
#' author: "Your Name"
#' format: html
#' ---

#' ## Setup

library(palmerpenguins)
library(ggplot2)

#' ## Bill Size Analysis
#'
#' @fig-bill-sizes shows the relationship between
#' bill length and depth.

#| label: fig-bill-sizes
#| fig-cap: "Bill dimensions by species"
#| warning: false

ggplot(penguins, aes(x = bill_length_mm, y = bill_depth_mm, color = species)) +
  geom_point() +
  theme_minimal()

You can render the script just like a .qmd file:

quarto render analysis.R
quarto preview analysis.R

This lets you keep your analysis as a plain R script that can also be sourced or run interactively.

7.4 Environment

7.4.1 How Quarto finds R

Quarto looks for an R installation in the following order:

  1. The QUARTO_R environment variable, if set
  2. The R executable found on your system PATH
  3. Platform-specific default locations (e.g., /usr/local/bin/R on macOS)

If Quarto is using a different R than you expect, you can override the lookup by setting QUARTO_R to the path of your preferred R binary. One way to do this is in a .env file in your project directory:

.env
QUARTO_R=/opt/R/4.4.0/bin/R

Quarto automatically reads .env files in the project root during rendering.

To see which R installation Quarto is currently using, run quarto check from the terminal:

quarto check knitr

The output will include the detected R version and path, along with the knitr and rmarkdown package versions, e.g.:

Quarto 1.10.3
[✓] Checking R installation...........OK
      Version: 4.6.0
      Path: /Library/Frameworks/R.framework/Resources
      LibPaths:
        - /Users/mine/Library/R/arm64/4.6/library
        - /Library/Frameworks/R.framework/Versions/4.6/Resources/library
      knitr: 1.51
      rmarkdown: 2.31

[✓] Checking Knitr engine render......OK

If the R path shown doesn’t match what you expect, that’s likely the source of any “package not found” or version mismatch issues.

7.4.2 Diagnosing environment issues

When Quarto renders a document, it starts a fresh R session. This session might use a different R installation or library paths than what you use interactively. In particular, packages you’ve installed in your interactive session may not be available if the rendering session uses different library paths, and environment variables like PATH may also differ.

Add a code cell to your document that outputs your session information:

```{r}
#| eval: false
sessionInfo()
.libPaths()
```

This will show the R version and the library paths used during rendering. For more detailed diagnostics, use the sessioninfo package:

```{r}
#| eval: false
sessioninfo::session_info(to_file = "quarto-session-info.txt")
```

Compare either output with your interactive R session to identify discrepancies. If library paths differ, that explains missing packages; if the R version differs, see Section 7.4.1 for how to control which R installation Quarto uses.

7.5 Multiple languages

While this chapter focuses on R, Quarto documents can include code cells from multiple languages. Each language runs in its own separate session—objects created in R are not directly accessible from Python, and vice versa.

```{r}
x <- 42
```
```{python}
# This cannot access the R variable x
y = 100
```

You might, for example, use R for statistical modeling and Python for certain visualizations.

TipSharing data between languages

To share data between R and Python cells, save intermediate results to files (e.g., CSV, Parquet) and read them in the other language. For tighter integration, consider the reticulate package, which allows R and Python to share objects directly.

If your document contains only R code cells, Quarto uses the knitr engine. If it contains only Python cells, it uses Jupyter. When both are present, Quarto uses the engine specified in your YAML header, or knitr by default.
